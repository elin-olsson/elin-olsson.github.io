Router Lockout Incident

During a routine security audit of a home network, ssh-tls-auditor unintentionally triggered a lockout on a primary gateway running Suricata in IPS mode. The incident provided direct insight into how inline intrusion prevention systems respond to audit traffic — and drove four versions of tool improvements.

1. The Incident

While running ssh-tls-auditor v1.7.1 against a home router, the tool initiated a rapid sequence of connection attempts to enumerate supported SSH key exchange algorithms, TLS cipher suites, and open ports. Within approximately 15 seconds, the router's management interface became unresponsive and all outbound traffic from the scanning host was dropped.

2. Root Cause Analysis

The lockout had two contributing factors:

  • Burst connection pattern: The auditor opens multiple parallel connections to probe algorithms and protocols. This produced a high-frequency burst that matched known scan signatures.
  • Suricata in inline IPS mode: The router was running Suricata with ET SCAN rules in inline (not detection-only) mode. Once the pattern was matched, Suricata dynamically injected iptables rules to block the source IP in real time.

Notably, the same scan against localhost or a local VM did not trigger the lockout — loopback traffic never passes through the physical network interface where Suricata sits inline.

3. Remediation — v1.7.2 through v1.7.4

The incident drove a series of targeted improvements to the auditor:

VersionChangePurpose
v1.7.2--delay SECONDS, --ssh-port PORTGroup-level delay between check modules; support for non-standard SSH ports
v1.7.3--skip-ping, external IP warning, ping pre-checkWarns before scanning public targets; skip ICMP check if filtered
v1.7.4--conn-delay SECONDSPer-connection delay applied before every socket call — finer-grained than --delay and more effective against IDS pattern matching

4. Lessons Learned

  • --delay is not enough against IPS: A group-level delay reduces burst peaks but doesn't prevent individual connections from arriving in a tight cluster. Per-connection delay (--conn-delay) spreads the traffic profile more evenly.
  • Suricata suppress list: The correct long-term fix for authorized testing is to add the scanner IP to Suricata's suppress list before scanning, not to slow down the tool.
  • Always test locally first: Running the tool against localhost or a local VM before targeting network devices avoids triggering IPS rules on production infrastructure.
