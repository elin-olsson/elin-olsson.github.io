ghost-trail — Technical Report

A Python-based DFIR (Digital Forensics & Incident Response) tool that reconstructs what happened on a Linux system after the fact. It parses binary login records, shell histories, recent file activity, and log gaps to produce a timestamped forensic timeline — exported as an interactive HTML report.

Core Functionality

1. Binary Login Records (wtmp/btmp)

Linux stores login and logout events in /var/log/wtmp and /var/log/btmp as fixed-size binary structs (utmp format). A custom parser reads these without relying on the last command, extracting user, terminal, source IP, and timestamps — even if the binary has been partially overwritten.

2. Shell History Analysis

The tool reads both ~/.bash_history and ~/.zsh_history. Zsh history files embed Unix timestamps in the format : 1714000000:0;command — these are stripped and converted to readable datetimes. The tool also scans for anti-forensic commands:

  • history -c — clears in-memory history
  • unset HISTFILE — disables history logging for the session
  • rm *log* — log deletion attempts

3. Timeline Construction

All events are merged into a single chronological timeline with three event types:

TypeColourExamples
ALERTRedAnti-forensic commands, deleted executables, suspicious file access
LOGINCyanSSH logins, console sessions, failed login attempts
FILEGreenRecently modified files, accessed configuration and credential paths

4. Log Gap Detection

The gap detector compares consecutive timestamps in system logs (auth.log, syslog). An unusually long silence — where no events were recorded — can indicate that logs were deleted or a system was powered off outside of normal patterns.

5. Interactive D3.js Timeline

The HTML report includes a D3.js timeline with three swim lanes (ALERT / LOGIN / FILE). Each event is represented as a coloured dot that can be hovered for details. The timeline supports zoom and pan, and filter checkboxes allow focusing on specific event types. Clicking an event scrolls to its entry in the event log below.

Technical Challenges & Solutions

  • Binary struct parsing: The utmp struct layout differs between Linux versions (32-bit vs 64-bit time fields). The parser handles both layouts and validates the record type field before accepting an entry.
  • PermissionError on CI: In GitHub Actions, path.exists() raises PermissionError for paths like /root/.bash_history — not just open(). The history parser wraps both calls in separate try/except blocks.
  • Zsh timestamp stripping: Zsh history entries have the format : 1234567890:0;actual command. A regex strips the timestamp prefix before the command is recorded in the timeline.
  • JS in HTML template: The D3 timeline is embedded as a plain string constant (not an f-string) with __PLACEHOLDER__ tokens replaced at export time — avoiding {{/}} escaping throughout several hundred lines of JavaScript.
